# MotionGame 项目结构说明文档

> 本文档用于说明 MotionGame 项目的整体软件架构、分层职责以及各模块之间的关系，便于后续维护、扩展和协作开发。

---

## 1. 项目目标

MotionGame 是一个基于 ESP32 的体感交互项目，核心目标是：

- 通过 **MPU6050 IMU** 获取姿态与动作数据
- 进行 **动作识别（倾斜 / 挥动 / 晃动）**
- 使用 **LCD** 进行实时可视化反馈
- 使用 **无源蜂鸣器** 输出音效反馈
- 采用 **非阻塞、分层清晰、可扩展** 的软件架构

---

## 2. 总体架构概览

项目采用自顶向下的分层结构：

```
MotionGame.ino
│
├── Input Layer        输入层（采集与更新）
│   └── input.*
│
├── Service Layer      服务层（算法 / 逻辑）
│   └── motion_service.*
│
├── Output Layer       输出调度层（更新驱动）
│   └── output.*
│
├── HAL / Driver       硬件抽象层
│   ├── buzzer.*
│   ├── mpu6050.*
│   └── lcd.*
│
├── Resource / Type    资源与类型定义
│   ├── audio_type.h
│   └── motion_type.h
│
└── UI Layer（可选）   显示逻辑
    └── ui.*
```

---

## 3. 各层职责说明

### 3.1 主程序层（MotionGame.ino）

**职责：**

- 系统初始化（setup）
- 主循环调度（loop）
- 连接各层模块，不实现具体业务细节

**特点：**

- 不直接访问硬件寄存器
- 不包含算法实现
- 只负责“调用顺序”和“模块协作”

---

### 3.2 Input Layer（输入层）

**文件：**
- `input.h / input.cpp`

**职责：**

- 对输入设备进行初始化
- 在 loop 中周期性更新输入数据

**当前包含：**

- MPU6050 采样与滤波更新

**设计原则：**

- 不做动作识别
- 不包含业务判断
- 仅负责“数据采集与刷新”

---

### 3.3 Service Layer（服务层）

**文件：**
- `motion_service.h / motion_service.cpp`

**职责：**

- 基于输入数据实现算法和逻辑
- 提供“事件级”或“语义级”结果

**当前实现：**

- 姿态计算结果读取
- 动作识别：
  - 倾斜（左 / 右）
  - 挥动（上 / 下）
  - 晃动
- 动作去重与冷却时间控制

**设计原则：**

- 不直接驱动输出设备
- 不关心 UI 或音效如何实现
- 返回“动作事件”供上层使用

---

### 3.4 Output Layer（输出调度层）

**文件：**
- `output.h / output.cpp`

**职责：**

- 对输出设备的 Update 进行统一调度

**当前包含：**

- Buzzer 的非阻塞更新驱动

**说明：**

Output 层本身不包含输出逻辑，仅作为 HAL 的更新入口，保证主循环结构清晰。

---

### 3.5 HAL / Driver Layer（硬件抽象层）

#### 3.5.1 蜂鸣器（buzzer.*）

**职责：**

- 无源蜂鸣器 PWM 驱动
- 连续音阶的非阻塞播放
- 音符状态机管理

**特性：**

- Play / Update / Stop / IsPlaying 接口
- 内部维护音符序列播放状态
- 与 Arduino-ESP32 3.x LEDC API 兼容

---

#### 3.5.2 MPU6050（mpu6050.*）

**职责：**

- IMU 原始数据读取
- 低通滤波
- 姿态解算（互补滤波）

**对外接口：**

- 加速度（g）
- 角速度（deg/s）
- 姿态角（deg）

---

#### 3.5.3 LCD（lcd.*）

**职责：**

- LCD 初始化与控制
- 提供基础绘制接口

**说明：**

LCD 层只提供“画什么”的能力，不包含 UI 逻辑。

---

### 3.6 Resource / Type Layer（资源与类型）

**文件：**
- `audio_type.h`
- `motion_type.h`

**职责：**

- 定义跨模块共享的数据结构
- 定义枚举、阈值、常量

**特点：**

- 不包含任何逻辑代码
- 只作为“公共协议”存在

---

### 3.7 UI Layer（可选）

**文件：**
- `ui.h / ui.cpp`

**职责：**

- 基于 LCD 绘制 UI 内容
- 示例：运行时间显示、简单界面

---

## 4. 主循环调用流程

```
loop():
  Input_Update();
  MotionService_Update();
  Audio_Play(若有动作事件);
  Output_Update();
```

**特点：**

- 所有模块均为非阻塞设计
- Update 驱动，利于扩展

---

## 5. 设计原则总结

- 分层清晰，职责单一
- HAL 不包含业务语义
- Service 不直接操作硬件
- 表驱动、状态机优先
- 避免 delay，使用 Update + millis

---

## 6. 后续可扩展方向

- 引入 GameService（动作 → 行为 → 反馈）
- 音效优先级与队列管理
- UI 状态机
- 多输入设备支持
